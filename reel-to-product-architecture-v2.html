S0

System overview

The product: a user watches a Reel, taps Share → your app, and 3–8 seconds later sees a card with (a) the exact product on Amazon.in and Flipkart, (b) 2–3 better-rated or cheaper alternatives, and (c) prices — every link affiliate-wrapped. Architecturally it is a five-hop pipeline, and each hop maps to one external service:

IngestReel URL → public image + captionMeta oEmbed APIApify / ScrapingBee scraper; user screenshot
ExtractImage + caption → structured product JSONGemini 2.5 Flash (vision)GPT-4o-mini / Claude Haiku

Design ruleNever let the app depend on a single vendor per hop. Every hop above has a documented fallback, because Instagram scraping breaks, PA-API accounts get suspended, and affiliate networks pause campaigns.
S1

Ingest — receive the link, get the media

1.1 Receive the share

Register your app as a share target so it appears in Instagram's share sheet. You receive plain text containing a URL like https://www.instagram.com/reel/DKx.../?igsh=...

  • Android: an intent-filter for ACTION_SEND with text/plain (and image/* to accept screenshots).
  • iOS: a Share Extension target; also register public.image so screenshots work.
  • Normalize: strip query params, resolve /share/ short links via one HEAD request, extract the shortcode (DKx...). The shortcode is your cache key for the whole pipeline.

1.2 Get image + caption — three routes, in order

RouteAPI / methodWhat you getCostRisk
A · OfficialMeta oEmbed API — graph.facebook.com/v23.0/instagram_oembed?url=...thumbnail_url, author, embed HTML for public posts/reelsFree (app token)Requires a Meta app + oEmbed App-Review approval; thumbnail only, no video file
B · ScraperApify Instagram Scraper actor, ScrapingBee, or Bright Data — fetch the post page, read og:image / og:descriptionFull-res image, caption, sometimes video URL≈ $0.5–2 per 1,000 postsAgainst IG ToS; blocks; use residential proxies; keep server-side only
C · ScreenshotUser shares a screenshot instead of a linkPixel-perfect frame incl. StoriesFreeNone — this is your ToS-clean guaranteed path

Ship A + C on day one; add B behind a feature flag. If the post is private or deleted, fall back to C with a friendly prompt ("share a screenshot instead").

1.3 Reels are videos — take frames

The oEmbed thumbnail is usually the product-bearing frame. When it isn't (talking-head intros), and you have the video file from route B, sample keyframes:

ffmpeg -i reel.mp4 -vf "select='gt(scene,0.3)',scale=768:-1" \
       -frames:v 5 -vsync vfr frame_%02d.jpg   # 3–5 scene-change frames, resized for the LLM

Extract — one vision-LLM call → product JSON

Do not build a CV pipeline at MVP. One multimodal model call handles detection, brand OCR from overlays, and attribute extraction. Force JSON output; temperature 0.

2.1 The extraction contract

# System prompt (abridged)
You identify the SINGLE most prominent purchasable product.
Return ONLY valid JSON matching the schema. Use null when unsure.

# Output schema
{
  "category":      "crossbody bag",
  "brand_guess":   "Caprese",          // from logo/overlay/caption, else null
  "product_name":  "mini quilted crossbody bag",
  "attributes":    ["black","faux leather","gold chain strap"],
  "gender_or_use": "women",
  "price_band_inr":"800-2500",
  "search_queries":["black quilted mini crossbody bag chain strap",
                    "caprese sling bag black",
                    "quilted sling bag women"]
}
GuardrailsReject frames where the model returns category:null or confidence is low — ask the user to tap the product in a frame picker instead of guessing. Wrong extraction poisons every later stage.
S3

Match — same product on Amazon & Flipkart

Two engines run in parallel: a visual engine (image in → matching listings out) and a text engine (extracted queries → store search). Merge their results before ranking.

3.1 Visual engine — Google Lens via SerpApi fastest path

Purpose
Find visually identical / near-identical listings across all retailers in one call
Endpoint
GET serpapi.com/search?engine=google_lens&url=<image_url>&country=in
Returns
visual_matches[] with title, link, source (amazon.in / flipkart.com / myntra.com...), price, thumbnail
Cost
Plan-based, roughly $0.005–0.01 per search at volume; SearchApi.io is an equivalent alternative vendor
Note
Pass a cropped product image (from Stage 2's frame) — full lifestyle frames dilute matches

3.2 Text engine per store

StoreAPIPurposeAccess reality
Amazon.inProduct Advertising API 5.0 · SearchItems / GetItemsOfficial search, title, price, rating, review count, ASIN, affiliate-ready URLsNeeds an Amazon Associates account; access is revoked without ~3 qualifying sales in 180 days — bootstrap with Rainforest first
Amazon.in (unofficial)Rainforest APIAmazon search + product data without PA-API approvalPaid (from ~$0.005–0.02/request); use until Associates is unlocked
FlipkartCuelinks Flipkart campaign + product feeds; Flipkart Affiliate API exists but only for already-registered affiliatesSearch/catalog + commissionable linksFlipkart stopped direct public affiliate sign-ups for new small affiliates — join through Cuelinks/EarnKaro instead
All storesSerpApi Google Shopping · engine=google_shopping&gl=inCross-store price comparison incl. Amazon, Flipkart, Myntra, AjioSame SerpApi account as Lens — one vendor, two engines
Merge logicDe-duplicate by normalized title + price proximity. A listing that appears in BOTH the Lens results and a store search is almost certainly the exact product — pin it to the top as "Exact match".
S4

Rank — exact match first, then genuinely better picks

4.1 Deterministic score (do NOT let the LLM invent numbers)

score = 0.35 * rating_norm        # rating / 5
      + 0.25 * log10(reviews+1)/5 # social proof, dampened
      + 0.25 * price_fit          # 1 - |price - band_mid| / band_mid, clamped 0..1
      + 0.15 * attribute_overlap  # matched attrs / total attrs

4.2 LLM re-rank (one cheap text call)

Send the top 10 scored listings (title, price, rating, reviews) back to Gemini Flash: "Pick the exact match if present, then the 3 best alternatives. For each alternative, one 8-word reason (better rated / cheaper / same specs)." The model writes the human-readable "why", never the ranking numbers.

Card layout1× Exact match (Amazon + Flipkart buttons side by side, prices visible) · 3× "Better picks" with the one-line reason · 1× "See 20 more" long-tail link. That card is the entire product.

S6

Backend — queue, cache, and the cost math

6.1 Shape

  • API server: FastAPI (Python) or Hono/Express (Node) on Railway / Fly.io / Cloudflare Workers.
  • Worker queue: the scan is async (3–8 s). Enqueue on share, push result via WebSocket/poll. Celery + Redis or BullMQ.
  • Postgres: posts(shortcode, media_url, caption, extract_json) · matches(post_id, store, url, price, rating, reviews, score) · clicks(match_id, user_id, ts) for affiliate attribution.
  • Redis: cache the finished card keyed by shortcode, TTL 24 h (also keeps you inside Amazon's data-freshness rules).
The economics of viralityViral reels are shared by thousands of users. With shortcode-level caching, user #2 to #10,000 of the same reel costs ₹0 in API spend and gets an instant result. Your marginal cost concentrates only on fresh content.

S8

Compliance & failure modes

  • Instagram ToS: scraping violates it. Keep route B server-side, low-volume, and lead with oEmbed + screenshots. Never ask users for their IG password.
  • Meta app review: oEmbed needs an approved use case — describe it as "preview of user-shared public links".
  • Amazon rules: price/data display must be fresh (re-query beyond 24 h), and the Associates account needs early sales — seed it with friends' purchases in week one.
  • Affiliate disclosure: show "links may earn us a commission" in the app (required by Amazon and Indian ASCI guidelines).
  • Privacy: you process only public posts and user-submitted screenshots; delete raw media after extraction; state it plainly — privacy is a selling point.
  • Failure UX: private post → screenshot prompt · no match → show category search on both stores anyway · API down → cached similar products.
S9

4-week MVP build order

  1. Week 1 — the spine: share extension → FastAPI endpoint → Gemini extraction → SerpApi Lens → raw JSON list on screen. No design, no cache.
  2. Week 2 — the card: merge + score + LLM re-rank; Amazon (Rainforest) + Flipkart (Cuelinks) buttons; Redis shortcode cache.
  3. Week 3 — money & polish: Cuelinks/EarnKaro wrapping, click logging, scan animation, screenshot fallback, empty states.
  4. Week 4 — distribution: Meta oEmbed review submission, Associates signup, and 10 demo TikToks/Reels of the share-to-card magic moment — the app IS the content.

Everything above is swappable by design: if one vendor breaks, its stage has a named replacement. Ship the spine first; optimize vendors with real traffic data.
